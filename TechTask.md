Реализуй фронтенд для кроссплатформенного веб-приложения на Vue + ionic + vite.
Актуальная aip-схема находится в @api_schema\\openapi.json.

# Layout
Базовый layout приложения зависит от платформы:
- на десктопе это шапка с названием и логотипом сервиса, навигацией и кнопкой входа/профиля с правого края
- на мобильных устройствах это шапка с кнопкой назад и BottomNav элемент (в общем современный стандарт мобильной разработке)

# Авторизация
Систем использует access-refresh токены
- access токен это jwt, который можно дефолтным образом расшифровать, получив информацию, которую можно отобразить
- access токен используется в Bearer заголовке
- access стоит хранить в in-memory переменной для максимальной безопасности: ситуаций, когда такая переменная слетает очень много, а достать ее очень тяжело
- refresh токен автоматический храниться в cookie с максимально жесткими настройками и автоматически отправляется при одном конкретном запросе с конкретным path

## Вход
Страница входа содержит карточку входа, с кнопкой перехода на регистрацию в верхнем правом углу

## Регистрация
Регистрация содержит аналогичную карточку с ссылкой на вход также в верхнем правом углу
Помимо этого содержит следующие пункты
- адрес почты
- username с автоматической проверкой занятости при остановке ввода (по аналогии с github)
- пароль и подтверждение пароля с проверками, отвечающими современным стандартам (также можно добавить шкалу сложности пароля)
- чекбокс об прочтении пользовательского соглашения и политикой конфиденциальности
Регистрационная карточка должна располагать всеми стандартными подсказками и алертами и прочим

## Верификация
После регистрации на ранее введенную почту автоматически отправляется письмо через соответсвующий эндпоинт, и открывается страница с соответсвующим сообшением кнопкой с таймором на полторы минуты для повторной отправки (значение таймера должно быть защищено от изменений и сбрасывания через обновление страницы и прочего)
и кнопкой перехода ко входу с соответсвующим параметром (который автоматически введет соответсвующую почту)

# Основной онтент
Приложение содержит 3 основных раздела (панели) и 2 технических дополнительных:
1. Главная
2. Для руководителей (не скрыт, но active только если к пользователю прикреплена персона, являющаяся руководителем хотя-бы одного коллектива)
3. Для админов (скрытый, если пользователь не является superuser, о чем информацию можно найти в bearer jwt или по эндпоинту users/me)
4. Профиль / входа (если нет авторизации)
5. Настройки (смена темы, информация о приложении, выбор формата дат и первого дня недели)

## Главная
На главной отображается календарь с мероприятиями и список мероприятий. Внешний вид элементов зависит от видимости (руководитель/участник) Внешний вид отличается в зависимости от платформы:
- на десктопе полноразмерный календарь слева, а список справа
- на мобильных устройства сначала идет мини-календарь, размером с 1 неделю, кнопка для раскрытия полноразмерного календаря (в качестве модального окна), и список мероприятий

Список мероприятий представляет из себя бесконечно прокручивающийся в обе стороны список из карточек EventPreview. Список отсортирован по дате. Календарь и список синхронизированы:
- если календарь переключается на конкретную дату, то список перематывается на эту дату
- если список пролистывается на определенную дату, то календарь переключается на нее
- если список пролистывается вверх, то запросом подгружается новая порция мероприятий с фильтром date__lte до минимальной даты, на которой приостановилась загрузка, вторым ограничением выступает не конкретная дата, а количество подгружаемых мероприятий (поскольку между мероприятиями может быть приличный перерыв).
- аналогично для пролистывания вниз
- после пролистывания проверяется, находиться ли крайний (в любую сторону) ивент за пределами видимости текущего состояния календаря, и если нет, то последующими запросами дозагружается информация по всем мероприятиями, которые могут расположиться на текущем виде календаря (недельного или полноразмерного)

Календарь всегда отображает текущую дату тонкой обводкой и обводкой с заливкой выбранную дату.
При первичной загрузке мероприятий, загружаются все мероприятия для заполнения текущего вида.
При дополнительных прогрузках мероприятий и перелистываниии календаря проверяется заполняемость календаря мероприятиями

Страница также содержит фильтры по коллективам 

Разница между руководителем и участником в видимости элементов:
- руководитель видит мероприятия любого статуса
- руководитель видит счетчик количества отметок для своего коллектива
- участник видит элементы управления своими записями участия для данного мероприятия 

### EventPreview
Элементы списка мероприятий это карточки со следующий обязательным содержимым:
- название
- дата (время, если имеется первая стадия)
- статус (для руководителя, желательно визуально а не текстом)
- чип/кнопка, отображающий значение is_attended с возможностью переключения по нажатию (если соответствующая запись attendance ещё не заверена; отображается, только если для персоны есть запись attendance); также данный чип может принимать состояние "требует отметки" (фиолетовый) для тех мероприятий attendance для которых у текущей персоны имеет поле edited_at === null (ни разу не изменялся), который по нажатию выставляет значение is_atended = true и меняет свое состояние на зелёный чип (чтобы поменять на красный нужно нажать ещё раз)
- чип с комментарием  (первые 16 символов; отображается только если у персоны есть запись attendance для этого мероприятия, не активен если эта запись заверена) по нажатию открывается модальное окно редактирования, чип содержит крестик для удаления комментария
- если соответствующая запись attendance уже заверена, то должен быть соответствующий чип, а чипы присутствия и комментария не дают эффекта при нажатии и визуально отображают свою не активность
- у руководителя отображается счётчик сколько отметок присутствия от общего количества истины
- Если персона относится к более чем одному коллективу (в качестве руководителя и/или в качестве участника) в рамках участия в одном и том же мероприятии, то на карточке соответствующие элементы отображаются в качестве списка с элементами, каждых из которых содержит в заголовке название соответствующего коллектива и всеми элементами управления/отображения; иначе все выглядит как список с одним единственным элементом

По нажатию на саму карточку открывается страница с подробной информацией о мероприятии

## Панель руководителя
В разделе руководителя есть подразделы с стандартным форматом для менеджмента: таблицы/списка, поиском, фильтрами и сортировкой
- участники
- роли
- мероприятия

В десктоп версии в шапке навигации пункт "Руководитель" содержит подпункты с коллективами, которыми руководит связанная с пользователем персона.
В мобильной версии есть два варианта как можно реализовать переключение между коллективами:
1. BottomNav элемент "руководитель" является выпадающим списком (должна присутсвовать соответсвующая стрелочка) при повторном нажатии на которых вверх будет выпадать список с коллективами
2. Заголовок панели должен содержать выпадающий список

В памяти должно храниться соответствие sub аккаунта (из jwt) и id последнего выбранного коллектива, чтобы пользователь не делал выбор каждый раз

## Панель администратора
Панель администратора содержит подразделы для менеджмента всех ресурсов, которые доступны в api
- пользователи
- персоны (включая контакты, также для персон, у которых нет прикрепленных пользователей должна быть активна кнопка для создания пригласительной ссылки с возможностью задать время действия)
- организации (включая факультеты, университеты и коллективы)
- коллективы (включая членов и роли)
- профили (включая студентов, групп для них и сопутствующие ресурсы)
- гео (адреса, места и тд)
- мероприятия (включая участия присутствия, ссылки, файлы и тд)

Навигацию по панели реализовать в качестве бокового меню, с разделением по микросервисам api.
Получается, если пользователь является суперюзером (в jwt sup = true) у него появляется левое боковое меню

## Панель аккаунта
В шапке справа присутсвует иконка профиля, по нажатию на которую открывается боковое меню с управлением аккаунтом (в том числе кнопка выхода)
Если пользователь не авторизован, вместо иконки там появляется кнопка для перехода на страницу входа

## Страница мероприятия
На страницу мероприятия разными способами может попасть как руководитель, так и участник. В зависимости от этого будет разная видимость.
Но базово это основная информация о самом мероприятиии, включая все его этапы (на десктопе слева) и список участвующих коллективов (на десктопе справа):
- каждый элемент списка это карточка, содержащая название коллектива и раскрывающийся список присутствий
- участник может управлять только своим присутсвием (если оно еще не заверено) и просматривать остальных
- руководитель имеет полное управление всеми присутствиями
- если пользователь не является ни участником ни руководителем коллектива, участвующего в мероприятии, то будет видна только общая информация 
- при это если пользователь является руководителем какого-то коллектива, то он может прямо оттуда создать участие своего коллектива
- пользватель может раскрыть присутсвия только тех коллективов, в которых является участником или руководителем
- такие коллективы находятся в начале списка
- если у пользователя есть attendance для определенного коллектива, то элемент выпадающего списка будет виден поверх данного списка (для быстрого доступа), то есть на карточке коллектива на странице мероприятия будет отображаться элемент с собственным attendace пользователя с возможность изменения (если не заверено)
- у руководителя присутствуют элементы управления пристусвиями (добавить, удалить, изменить, заверить, заверить все)
- элементы изменения attendace у руководителя имеют схожую логику работы как и на EventPreview (чипы с переключением по нажатию)

## Создание мероприятия / участия

Модальное окно создания мероприятия / участия для руководителя коллектива является формой для заполнения:
- В начале можно выбрать мероприятие со статусом "template" что автоматически перенесет с шаблона все поля, кроме даты (включая присутствия и стадии)
- выбор даты после выбора шаблона активирует перерасчет с относительным смещением всех временных меток перенесенных с шаблона стадий мероприятия (например если в шабоне стояла дата 01.01.2026 и у стадии временная метка стояла 2026-01-02 12:40 а при выборе шаблона пользователь выбрал 05.07.2026, то у 1 стадии временная метка поменяется на 2026-07-06 12:40)
- поле со статусом по умолчанию имеет значение "draft", но если все поля заполняются, то автоматически меняется на "active" но только если пользователь не трогал селектор статуса мероприятия (для большей интуитивности селектор в автоматическом режиме можно выделить визуально)
- в форме присутсвуте раздел "заявленные участники" для выбора перечня участников, для которых будет создан attendace, по умолчанию он будет неактивен и на не будет надпись, что сейчас заявлены все n активных участников коллектива и кнопка для настройки; при нажатии появляется список, поисковая строка и чипы-фильтры с ролями, работающие по принципу togle (Одно нажатие зажигает чип, и все участники данной роли добавляются, второе нажатие гаснет и все участники данной роли удаляются), также дополнительно чип "все" чтобы добавить/убрать всех участников вне зависимости от роли. Если все участники данной роли выбраны, то чип должен автоматически загораться 
- Также вместо шаблона можно выбрать существующее мероприятие, чтобы создать участие коллектива в нем, тогда форма сократиться до выбора участников для создания attendance
